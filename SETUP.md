# 認証フェーズ 設定手順書（サトシさん用）

アプリ本体（`index.html`）の実装は完了しています。
ただし、**Firebase側の設定が2つ**と、**GitHub Pagesの公開設定**は
Firebaseコンソール・GitHubの画面からしか変更できないため、以下の手順をお願いします。
（どれも5分程度で終わります）

---

## 手順1：匿名ログインを有効にする（作業員参加に必要）

作業員は「現場コード＋名前」だけで参加します。このとき裏側では
Firebaseの「匿名ログイン」という仕組みを使うため、有効化が必要です。

1. https://console.firebase.google.com を開き、プロジェクト **genbakanri-d811d** を選ぶ
2. 左メニューの **構築 → Authentication** をタップ
3. 上のタブから **Sign-in method（ログイン方法）** を選ぶ
4. プロバイダ一覧の **匿名** をタップ
   - 一覧に無い場合は「新しいプロバイダを追加」から「匿名」を選ぶ
5. **有効にする** をオンにして **保存**

これをしないと、作業員が参加ボタンを押したときに
「作業員参加の準備がまだできていません」というエラーが出ます。

## 手順2：Firestoreのセキュリティルールを設定する

今は「テストモード（誰でも読み書きできる）」のままです。
このリポジトリにある **`firestore.rules`** の中身をコンソールに貼り付けて、
「自分の現場だけ読み書きできる」状態にします。

1. Firebaseコンソール → **構築 → Firestore Database**
2. 上のタブから **ルール** を選ぶ
3. 表示されているルールを**全部消して**、`firestore.rules` ファイルの中身を**丸ごと貼り付ける**
4. 右上の **公開** ボタンを押す

ルールの内容（ざっくり）：
- 職長：自分が作った現場（ownerUidが自分）だけ読み書きできる
- 作業員：参加した現場だけ、作業リストの受け取りと完了報告の送信ができる
- 昔のデータ（genba/main）：読み取り専用で残してある（新しい現場を作るときの「取り込み元」）
- それ以外の場所は誰も読み書きできない

## 手順3：GitHub Pagesで公開する

1. GitHubで `satoshimr929-cyber/koutei` リポジトリを開く
2. このブランチ（`claude/code-handoff-review-7efgxr`）のプルリクエストを確認して、
   問題なければ **main（デフォルトブランチ）にマージ** する
3. リポジトリの **Settings → Pages** を開く
4. 「Build and deployment」の **Source** を **Deploy from a branch** にする
5. **Branch** を `main`（フォルダは `/ (root)`）にして **Save**
6. 数分待つと `https://satoshimr929-cyber.github.io/koutei/` で開けるようになります

※ 既存の `satoshimr929-cyber.github.io` リポジトリ（別アプリ）には一切触れていません。
`koutei` は独立したリポジトリなので、既存アプリのURLもコードもそのままです。

### 公開後にもう1つ（Googleログイン用）

GitHub Pagesで公開したページからGoogleログインを使うには、
Firebaseに「このドメインからのログインを許可する」設定が必要です。

1. Firebaseコンソール → **Authentication → Settings（設定）** タブ
2. **承認済みドメイン** を開く
3. **ドメインを追加** で `satoshimr929-cyber.github.io` を追加

## 手順4：実機での動作確認（公開後）

以下の順で確認してください。

1. **職長ログイン**：スマホで `https://satoshimr929-cyber.github.io/koutei/` を開く
   → 入口画面が出る → 「職長としてログイン」→ メール＋パスワード（またはGoogle）でログイン
2. **現場を作る**：「＋新しい現場を作る」→ 現場名に「氷川台」など入力
   → **「これまでのクラウドデータを取り込む」にチェックが付いていることを確認**（これが氷川台データの移行です）
   → 「現場を作る」→ 現場コードとQRが表示される
3. **現場をひらく**：部屋・工程・進捗が今まで通り表示されるか確認
4. **作業員を招待**：メニュー（☰）→「現場コード・QR招待」→ QRを別のスマホで読み取る
5. **作業員参加**：読み取った側に参加画面が出て、現場コードが自動入力されている
   → 名前を入れて「参加する」→ 作業員のシンプル画面になる
6. **双方向のやり取り**：
   - 職長：メニュー→「作業リストを作る」→ クラウドで送る
     - 一番上の「渡す相手」で作業員を指定すると、その人にだけ届きます（全員向けも選べます）
   - 作業員：「作業リストを取り込む」→ 作業して完了チェック → 「完了を職長に送る」
   - 職長：メニュー→「完了報告を確認」→ 報告を開いて本体に反映（誰からの報告かも表示されます）
   - 職長：メニュー→「参加している作業員」→ 誰がいつ参加・ログインしたかを確認
     （「外す」でその作業員をこの現場から外せます）
   - 職長：メニュー→「送った作業リスト」→ クラウド上のリストを確認・削除
     （間違えて送ったリストを消せます。作業員が取り込み済みの端末には影響しません）
7. 全部OKなら、移行が成功しているので、旧データ（Firestoreの `genba` コレクション）は
   しばらく残したままでOK（ルール上、書き込み不可の読み取り専用になっています）。
   数週間問題なく使えたら、コンソールから手動で削除しても構いません。

## 手順5：Storageのセキュリティルールを設定する（写真共有フェーズ）

写真はFirebase Storageに保存されるようになりました。今は「本番環境モード（全拒否）」なので、
このリポジトリの **`storage.rules`** の中身を貼り付けて、写真の読み書きを許可します。

1. Firebaseコンソール → **構築 → Storage**
2. 上のタブから **ルール** を選ぶ
3. 表示されているルールを**全部消して**、`storage.rules` ファイルの中身を**丸ごと貼り付ける**
4. 右上の **公開** ボタンを押す

ルールの内容（ざっくり）：
- 写真のアップロード・削除：その現場を作った職長だけ
- 写真の表示：ログイン済みの人（職長・作業員）
- 画像以外・2MB以上のファイルはアップロード不可
- 写真置き場以外の場所は全部禁止

### 写真共有の動作確認

1. 職長：部屋を開いて写真を追加 → 「写真を保存中…」と出て追加される
2. 職長：メニュー→「写真をクラウドへ移行」→ 端末に残っている古い写真をまとめてクラウドへ
   （移行しないと、その写真は作業員からは見えません）
3. 職長：作業リストを作って送る → 作業員：取り込むと、部屋の**参考写真がスマホで見える**（タップで拡大）
4. 写真を削除すると、クラウド側の写真も一緒に消えます（容量のムダ・課金を防ぐため）
5. 電波が悪くてアップロードに失敗した写真は端末内に保存され、
   「写真をクラウドへ移行」でいつでも再送できます

## 困ったとき

- **入口画面を止めたい（今まで通りに戻したい）**：
  `index.html` の上の方にある `window.AUTH_GATE_ENABLED = true;` を `false` に変えると、
  入口画面なしの従来動作に戻ります（緊急避難用）。
- **「作業員参加の準備がまだできていません」エラー** → 手順1（匿名ログイン）が未設定です
- **「権限がありません」エラー** → 手順2のルールが正しく貼り付いているか確認してください
- **Googleログインでポップアップが閉じてしまう** → 手順3の「承認済みドメイン」を確認してください
